# Agent Development Rules – Smart Travel Ecosystem

## 1. Purpose

This document defines strict operational rules for all AI agents and code generation tools used in this project.

The objective is to ensure:

* Consistency with defined architecture
* Zero hallucination
* Strict adherence to schemas and user instructions
* Predictable and correct outputs

---

## 2. Source of Truth

The agent must always treat the following as the only valid references:

* Database schemas defined in the project
* Microservice boundaries defined in the architecture
* Folder structure and service naming conventions
* Explicit user instructions in the prompt

If there is any conflict:

1. User prompt takes priority
2. Then schema definitions
3. Then architecture rules

---

## 3. Prompt Adherence Rules

The agent must:

* Only generate output based on the user’s request
* Not add extra features, assumptions, or enhancements unless explicitly asked
* Not skip required parts mentioned in the prompt
* Not generalize when specific implementation is requested

The agent must not:

* Expand scope beyond the given prompt
* Introduce unrelated logic
* Modify requirements silently

---

## 4. Schema Enforcement Rules

The agent must:

* Use only the tables, fields, and structures defined in the schema
* Maintain exact field names and data types
* Respect service boundaries (each service is independent)
* Avoid cross-service joins and foreign key assumptions

The agent must not:

* Create new schema elements unless explicitly requested
* Rename fields or change types
* Merge multiple services into one
* Infer relationships not defined

---

## 5. Microservice Boundaries

Each service must remain isolated.

The agent must:

* Treat each service as independently deployable
* Communicate between services only via APIs or events
* Avoid direct database access across services

The agent must not:

* Create tight coupling between services
* Share internal service logic across boundaries
* Use direct table joins across services

---

## 6. Frontend Rules

The agent must:

* Follow component-based architecture
* Use defined API services for backend communication
* Maintain separation between UI, state, and API layers

The agent must not:

* Embed backend logic in frontend components
* Hardcode business logic that belongs to backend services

---

## 7. Backend Rules

The agent must:

* Follow service-based architecture
* Keep controllers, services, and data layers separate
* Validate inputs before processing
* Handle errors explicitly

The agent must not:

* Mix multiple service responsibilities
* Bypass validation or error handling
* Directly manipulate unrelated service data

---

## 8. Database Rules

The agent must:

* Keep schemas normalized and consistent
* Use clear naming conventions
* Maintain audit fields where required (created_at, status)

The agent must not:

* Introduce unnecessary complexity
* Create implicit relationships across services
* Store redundant or duplicated data without reason

---

## 9. Workflow for Task Execution

For every user request, the agent must follow this sequence:

1. Understand the exact user requirement
2. Identify the relevant layer:

   * Frontend
   * Backend
   * Database
3. Map the request to existing schemas and services
4. Generate only the required output
5. Validate consistency with:

   * Schema
   * Architecture
   * Prompt constraints
6. Return structured and clean output

---

## 10. Output Standards

All outputs must be:

* Structured and readable
* Minimal but complete
* Aligned with project architecture
* Free from assumptions

---

## 11. Error Handling and Uncertainty

If the request is unclear or incomplete:

* The agent must not guess
* The agent must either:

  * Ask for clarification, or
  * Provide a minimal valid interpretation

---

## 12. Success Criteria

An output is considered correct only if:

* It strictly follows the user prompt
* It aligns with the defined schemas
* It respects microservice boundaries
* It does not introduce hallucinated elements
* It is implementable without modification

---

## 13. Enforcement Rule

If any rule in this document is violated, the output is invalid and must be regenerated.
