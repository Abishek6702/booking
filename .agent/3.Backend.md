# Backend Development Rules – Smart Travel Ecosystem

## 1. Purpose

This document defines strict rules for backend development and AI-assisted code generation.

The goal is to ensure:

* Strict adherence to microservice architecture
* Alignment with defined database schemas
* No hallucinated APIs or business logic
* Clean, scalable, and maintainable backend services

---

## 2. Scope

These rules apply to all backend services:

* Auth Service
* User Service
* Booking Service
* Payment Service
* Stay Service
* Vehicle Service
* Attraction Service
* Review Service
* Notification Service
* Analytics Service

---

## 3. Source of Truth

Backend must strictly follow:

1. Database schemas (defined per service)
2. Microservice boundaries
3. API contracts
4. User requirements from prompt

If conflict occurs:

1. User prompt (if explicit)
2. Schema definitions
3. Service architecture rules

Backend must NOT:

* Invent new fields not in schema
* Modify schema structure silently
* Assume relationships not defined

---

## 4. Microservice Architecture Rules

### 4.1 Service Isolation

Each service must:

* Own its own database
* Operate independently
* Expose functionality only via APIs/events

Services must NOT:

* Directly access another service’s database
* Share internal logic across services
* Create tight coupling

---

### 4.2 Communication Rules

Allowed communication methods:

* REST APIs (synchronous)
* Message Queue (asynchronous events)

Examples:

* BOOKING_CREATED
* PAYMENT_SUCCESS
* USER_REGISTERED

Services must NOT:

* Call internal DB of another service
* Bypass API gateway

---

## 5. API Design Rules

Each service must:

* Follow RESTful conventions
* Use clear and consistent endpoints
* Validate all inputs
* Return structured JSON responses

Example:

* POST /bookings
* GET /stays
* POST /payments

Response format must be consistent:

* status
* data
* error (if any)

---

## 6. Controller-Service-Repository Pattern

Backend must follow layered structure:

* Controller → Handles request/response
* Service → Business logic
* Repository → Data access

Controllers must NOT:

* Contain business logic

Services must:

* Handle validation, orchestration, decisions

Repositories must:

* Only interact with database

---

## 7. Booking Logic Rules

Booking Service must:

* Validate availability via respective service
* Prevent duplicate bookings
* Maintain booking status lifecycle:

  * PENDING → CONFIRMED → CANCELLED

Must NOT:

* Assume availability without verification
* Confirm booking before payment success

---

## 8. Payment Rules

Payment Service must:

* Process payments securely
* Store transaction status
* Support states:

  * SUCCESS
  * FAILED
  * PENDING

Must NOT:

* Fake payment success
* Store sensitive card data in plain text

---

## 9. Data Validation Rules

All inputs must be validated:

* Required fields
* Data types
* Business constraints

Must NOT:

* Accept invalid or incomplete data
* Rely only on frontend validation

---

## 10. Error Handling Rules

Backend must:

* Return proper HTTP status codes
* Provide meaningful error messages
* Log all failures

Must NOT:

* Return generic or unclear errors
* Crash on invalid input

---

## 11. Security Rules

Backend must:

* Use JWT authentication
* Enforce role-based access control
* Encrypt sensitive data

Must NOT:

* Expose internal APIs
* Leak sensitive information
* Allow unauthorized access

---

## 12. Logging and Monitoring

Each service must:

* Log requests and responses
* Track errors and performance
* Send metrics to analytics service

---

## 13. Transaction and Consistency Rules

Backend must:

* Ensure consistency across services using events
* Handle retries for failed operations

Example:

* Booking created → Payment triggered
* Payment success → Booking confirmed

Must NOT:

* Leave system in inconsistent state

---

## 14. Workflow for Task Execution

For every request:

1. Identify service responsible
2. Validate input
3. Apply business logic
4. Interact with database
5. Trigger events (if needed)
6. Return structured response

---

## 15. Output Standards

All backend outputs must be:

* Deterministic
* Schema-aligned
* Free from assumptions
* API-consistent

---

## 16. Success Criteria

Backend implementation is correct only if:

* It follows microservice boundaries
* It uses only defined schemas
* It validates all inputs
* It maintains correct data flow
* It produces consistent API responses

---

## 17. Enforcement Rule

If any backend code:

* Violates schema
* Breaks service isolation
* Introduces hallucinated logic

It must be rejected and regenerated.
