# Database Rules – Smart Travel Ecosystem

## 1. Purpose

This document defines strict rules for database design, schema usage, and data handling.

The goal is to ensure:

* Strict adherence to defined schemas
* No unauthorized schema changes
* Data consistency across microservices
* Prevention of hallucinated tables or fields

---

## 2. Scope

These rules apply to all databases across services:

* User Service Database
* Booking Service Database
* Payment Service Database
* Stay Service Database
* Vehicle Service Database
* Attraction Service Database
* Review Service Database
* Analytics Service Database

Each service must maintain its own independent database.

---

## 3. Source of Truth

The database schema is the final authority.

The system must strictly follow:

1. Defined SQL schemas
2. Table structures and field definitions
3. Data types and constraints

If conflict occurs:

1. Schema definition takes highest priority
2. Then backend logic
3. Then frontend usage

The system must NOT:

* Add fields not defined in schema
* Rename existing columns
* Change data types without explicit instruction

---

## 4. Schema Integrity Rules

The database must:

* Maintain exact table structures
* Preserve field names and types
* Enforce constraints (ENUM, CHECK, UNIQUE)

The system must NOT:

* Create duplicate tables
* Modify schema implicitly
* Introduce unnecessary fields

---

## 5. Microservice Database Isolation

Each service must:

* Own its own database
* Manage its own schema
* Operate independently

The system must NOT:

* Use foreign keys across services
* Perform cross-service joins
* Share direct database access

Communication must happen via:

* APIs
* Events

---

## 6. Table Design Rules

Each table must:

* Have a primary key
* Use clear naming conventions
* Include timestamps where needed (created_at)

Optional but recommended:

* status fields for lifecycle tracking

The system must NOT:

* Use ambiguous column names
* Store unrelated data in the same table
* Overload tables with multiple responsibilities

---

## 7. Data Consistency Rules

The database must:

* Maintain valid state transitions
* Ensure no orphan or invalid data
* Keep data aligned with business logic

Example:

* Booking must not be CONFIRMED without payment success
* Payment must reference a valid booking_id

The system must NOT:

* Allow inconsistent states
* Skip validation rules

---

## 8. ENUM and Constraints Rules

ENUM values must:

* Match exactly with defined schema
* Not be extended without explicit instruction

CHECK constraints must:

* Be respected during inserts and updates

The system must NOT:

* Insert invalid ENUM values
* Bypass constraints

---

## 9. Data Validation Rules

Before inserting/updating:

* Validate required fields
* Validate data types
* Validate business constraints

The system must NOT:

* Insert null or invalid data where not allowed
* Rely solely on frontend validation

---

## 10. Transactions and Atomicity

For critical operations:

* Use transactions where needed
* Ensure atomic operations

Example:

* Booking + Payment must be consistent

The system must NOT:

* Leave partial updates
* Create inconsistent data states

---

## 11. Logging and Audit Fields

Tables should include:

* created_at (mandatory)
* updated_at (if needed)
* status (where applicable)

The system must:

* Track important changes
* Maintain auditability

---

## 12. Performance Rules

The database must:

* Use indexing for frequent queries
* Optimize read/write operations

The system must NOT:

* Perform unnecessary heavy queries
* Fetch unused data

---

## 13. Data Access Rules

Data must be accessed:

* Only through respective service
* Through defined repositories or queries

The system must NOT:

* Allow direct access from other services
* Expose raw database externally

---

## 14. Workflow for Data Operations

For every database operation:

1. Validate input
2. Match with schema
3. Apply constraints
4. Perform insert/update/delete
5. Return structured result

---

## 15. Success Criteria

Database usage is correct only if:

* Schema is unchanged and respected
* Data remains consistent
* No cross-service violations occur
* All constraints are enforced
* No hallucinated fields or tables exist

---

## 16. Enforcement Rule

If any operation:

* Violates schema
* Breaks constraints
* Introduces invalid data

It must be rejected immediately.
